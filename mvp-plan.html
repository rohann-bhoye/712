BucketDev · MVP Pivot

Edit your repo from your phone. No laptop.

A decision record and step-by-step plan for dropping the Docker sandbox in favor of a pure GitHub-API MVP that actually runs on Vercel — matching the original "mobile-first dev agent" goal.

!The Problem

Why "no laptop" wasn't actually true

src/lib/dev-workspace-docker.ts shells out directly to the docker CLI (docker run, docker exec) to clone the repo, install deps, build, test, and preview the app inside a container for every AI edit. That only works on a machine that has a Docker daemon running — it cannot run on Vercel serverless functions. So the backend could only ever run on a laptop, defeating the entire point of a phone-only editor.

The decision

Drop the Docker sandbox entirely. The MVP edit loop becomes pure GitHub API read/write — fetch file → AI generates new content → user reviews on phone → approve → commit via the GitHub Data API. No build, no test, no execution environment, nothing running anywhere. That's what makes the backend deployable on Vercel and the "no laptop" goal real.

3What Changes, File by File

  • Remove the Docker sandbox delete

    Delete src/lib/dev-workspace-docker.ts and the /api/dev/workspaces/[id]/preview route. Strip the container calls and building/testing states out of src/lib/dev-agent-runner.ts.

  • Multi-file, real-context AI edits new

    fetchRepoTree() in src/lib/dev-github.ts gives the AI the actual file list instead of hardcoding README.md. The AI can now return edits for multiple real files.

  • Real diffs instead of raw JSON fix

    Store { before, after, patch } per file using the diff npm package. approve/route.ts pushes .after; the phone renders .patch with +/− coloring instead of dumping the whole file.

  • Mobile chat as a step timeline redesign

    job.events already stores every step — render each one as its own row (Analyzing → Planning → Editing → Reviewing) instead of a single progress bar, then big Approve/Reject buttons at the decision point.

  • Phone reaches a deployed backend, not just localhost fix

    mobile/src/api.ts's BASE_URL becomes EXPO_PUBLIC_API_URL-driven so the app can point at the real Vercel deployment from anywhere.

  • Close the hardcoded-secret gap security

    src/lib/auth.ts and src/lib/secrets.ts stop silently falling back to a known hardcoded key — now that this is genuinely deployable, that's a real risk instead of a local dev nuisance.

4Accepted Tradeoff

Before (Docker)After (MVP)
Build + test run automatically before every commitNo verification — commit lands as soon as you approve
Live preview URL of the running appNo preview — read the diff, trust the repo's own CI
Only works with a laptop running DockerWorks from anywhere, backend lives on Vercel

If the AI writes broken code, you'll find out from the target repo's own GitHub Actions CI (if it has any) after the commit — not before. That's the deliberate simplicity tradeoff for this MVP.
